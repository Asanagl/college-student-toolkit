# CLAUDE.md — Agent 集群工程工具组配置

> 版本: v1.0  
> 用途: 多领域工程任务自动化协作  
> 核心机制: Orchestrator-Worker + 重新思考 Hook  
> 配套配置: `.claude/settings.json` (hooks 配置)

---

## 项目概述

本项目是一个基于 Agent 集群的 AI 工程工具组，通过多个专业 Agent 协作完成复杂工程任务。采用 Orchestrator-Worker 模式，支持自动化的代码生成、审查、测试和文档输出。

## 技术栈

- **协调框架**: Claude Code Agent 集群
- **通信协议**: 结构化 JSON 消息
- **质量门禁**: 代码审查 + 安全审计 + 测试覆盖
- **重试机制**: 重新思考 Hook (详见 `.claude/settings.json`)

## 代码生成规则

### 通用规范
- 函数长度不超过 50 行
- 必须包含输入校验和异常处理
- 注释说明"为什么"而非"做什么"
- 使用类型注解（TypeScript/Python）

### 架构设计规则
- 禁止过度设计，简单需求对应简单架构
- 技术选型必须提供至少 2 个备选方案对比
- 必须识别至少 3 个潜在风险
- 禁止在架构阶段输出具体实现代码

### 代码实现规则
- 遵循 SOLID 原则
- 敏感信息（密钥、密码）必须使用环境变量
- 数据库查询必须防 SQL 注入
- 用户输入必须做 XSS 过滤

### 测试规则
- 单元测试分支覆盖率 >= 80%
- 必须包含至少 1 个负面测试用例
- 测试用例必须独立，不依赖执行顺序
- 使用 Mock 隔离外部依赖

## Agent 角色定义

### 1. 需求解析 Agent (ReqParser)

**职责**: 将用户自然语言请求转化为结构化任务描述

**输入**: 用户自然语言
**输出**: 结构化 JSON

```json
{
  "task_type": "代码生成|架构设计|Bug修复|性能优化|文档生成",
  "domains": ["前端", "后端"],
  "tech_stack": ["React", "Node.js"],
  "complexity": "简单|中等|复杂",
  "requirements": ["功能点1", "功能点2"],
  "constraints": ["响应时间<200ms", "支持1000并发"],
  "ambiguous_points": ["未明确项"]
}
```

**规则**:
- 存在歧义时在 `ambiguous_points` 列出，不擅自假设
- 请求模糊时输出"需要澄清"并列出具体问题
- 检测到用户否定关键词时标记 `user_negation_detected: true`

### 2. 协调者 Agent (Orchestrator)

**职责**: 制定执行计划、调度 Agent、聚合结果、处理冲突

**输入**: 需求解析 Agent 的结构化输出
**输出**: 执行计划 DAG

```json
{
  "execution_plan": [
    {"agent": "架构设计Agent", "order": 1, "depends_on": []},
    {"agent": "代码生成Agent", "order": 2, "depends_on": ["架构设计Agent"]},
    {"agent": "代码审查Agent", "order": 3, "depends_on": ["代码生成Agent"]}
  ],
  "parallel_groups": [["代码审查Agent", "安全审计Agent"]],
  "termination_condition": "代码审查通过且安全审计无高危漏洞"
}
```

**规则**:
- 不得跳过代码审查 Agent
- 架构冲突时要求重试
- 收到否定反馈时触发重新思考 Hook

### 3. 架构设计 Agent (Arch)

**职责**: 系统架构设计、技术选型、接口契约

**输入**: 结构化任务描述
**输出**: 架构文档 + 接口定义

**必须包含**:
- 架构图描述（文本形式）
- 模块清单与职责
- 接口定义（路径、方法、参数、返回值）
- 数据模型定义
- 技术选型对比（>=2 个备选方案）
- 潜在风险清单（>=3 个）

**禁止**:
- 输出具体实现代码
- 过度设计

### 4. 代码生成 Agent (CodeGen)

**职责**: 根据架构设计生成可运行代码

**输入**: 架构设计文档 / 简单需求
**输出**: 代码文件 + 依赖清单 + 测试框架

```json
{
  "files": [
    {
      "path": "src/auth/login.js",
      "content": "// 代码内容",
      "explanation": "该文件负责..."
    }
  ],
  "dependencies": ["express", "jsonwebtoken"],
  "test_files": [...]
}
```

### 5. 代码审查 Agent (CodeReview)

**职责**: 审查代码质量

**审查维度**:
1. 安全性（SQL 注入、XSS、硬编码密钥等）
2. 性能（N+1 查询、内存泄漏等）
3. 可维护性（SOLID、耦合度）
4. 正确性（逻辑、边界条件）
5. 规范（命名、注释、格式）

**输出**:
```json
{
  "pass": false,
  "issues": [
    {
      "severity": "严重|警告|建议",
      "file": "src/auth.js",
      "line": 15,
      "description": "JWT密钥硬编码",
      "suggestion": "使用环境变量读取密钥"
    }
  ],
  "summary": "发现1个严重问题，必须修复后才能通过"
}
```

**规则**:
- `severity: "严重"` 时 `pass` 必须为 `false`，触发重新思考
- 给出具体修改建议，而非仅指出问题

### 6. 测试生成 Agent (TestGen)

**职责**: 生成完整测试用例

**测试类型**:
- 单元测试（正常路径、边界、异常）
- 集成测试（多模块交互）
- 性能测试（有性能约束时）

**要求**:
- 分支覆盖率 >= 80%
- 至少 1 个负面测试用例
- 测试用例独立，不依赖顺序
- Mock 隔离外部依赖

### 7. 文档生成 Agent (DocGen)

**职责**: 生成技术文档

**文档类型**:
- API 文档（接口、参数、返回值、错误码）
- 部署文档（环境、配置、启动、故障排查）
- 使用说明（用户视角操作指南）

**规范**:
- Markdown 格式
- 包含请求/响应示例
- 错误码完整列出
- 部署文档含故障排查

### 8. 性能优化 Agent (PerfOpt)

**职责**: 识别和修复性能瓶颈

**分析维度**:
- 时间/空间复杂度
- 数据库查询优化（索引、连接、子查询）
- 缓存策略和异步处理
- 性能基准测试方案

### 9. 安全审计 Agent (SecAudit)

**职责**: 安全漏洞检测

**检测范围**:
- OWASP Top 10
- 认证授权机制
- 输入验证和输出编码
- 敏感数据处理和传输安全

**规则**:
- `severity: "高危"` 必须触发重新思考，阻断发布
- 提供具体修复代码示例

## 工作流定义

### 简单任务（单函数/单文件）

```
用户请求 → 需求解析 → 协调者 → 代码生成 → 代码审查 → 输出
```

### 中等任务（模块开发）

```
用户请求 → 需求解析 → 协调者 → 架构设计 → 代码生成 → [代码审查 + 安全审计] → 测试生成 → 文档生成 → 输出
```

### 性能敏感任务

```
用户请求 → 需求解析 → 协调者 → 代码生成 → 代码审查 → 性能优化 → 代码生成(修复) → 代码审查(回归) → 输出
```

## 质量门禁

| 门禁项 | 通过标准 | 失败处理 |
|--------|---------|---------|
| 代码审查 | 无严重问题 | 触发重新思考 Hook |
| 安全审计 | 无高危漏洞 | 触发重新思考 Hook，阻断流程 |
| 测试覆盖 | 分支覆盖率 >= 80% | 触发代码生成 Agent 重新思考 |
| 性能指标 | 达到需求约束 | 触发性能优化 / 代码生成重新思考 |
| 架构冲突 | 模块接口匹配 | 触发架构设计 Agent 重新思考 |

## 通信契约

Agent 间通信标准消息格式:

```json
{
  "message_id": "uuid",
  "from_agent": "代码生成Agent",
  "to_agent": "代码审查Agent",
  "task_id": "task_uuid",

  "payload": { },
  "metadata": {
    "attempt": 1,
    "chain_depth": 2,
    "original_requirement": "开发用户认证模块"
  }
}
```

## 重新思考触发条件

当下游 Agent 对上游输出给出否定反馈时，自动触发重新思考:

| 触发场景 | 否定信号 | 目标上游 Agent |
|---------|---------|--------------|
| 代码审查不通过 | `pass: false` + 严重问题 | 代码生成 Agent |
| 安全审计高危漏洞 | `severity: "高危"` | 代码生成 Agent |
| 架构模块冲突 | 接口不匹配 | 架构设计 Agent |
| 测试覆盖率不足 | `< 80%` | 代码生成 Agent |
| 性能不达标 | 差距 > 50% | 代码生成 / 性能优化 Agent |
| 用户直接否定 | 否定关键词 | 最近一次输出 Agent |

**重试规则**: 最多 3 次 → 第 3 次仍失败 → 协调者标记阻塞 → 人工介入

**详见**: `.claude/settings.json` 中的 `hooks` 配置

## 用户否定关键词检测

以下关键词触发 `user_negation_detected: true`:

**强否定**（立即触发）:
- "不对" / "错了" / "不正确" / "有问题"
- "重新来" / "重来" / "再来一次" / "重做"
- "不是这样" / "不是我要的" / "不符合" / "偏离了"
- "完全错误" / "全错了" / "根本不对"

**弱否定**（结合上下文）:
- "改一下" / "修改" / "调整" / "优化一下"
- "太复杂了" / "太简单了" / "不够" / "多了"
- "能不能换一种" / "还有别的方案吗"
- "感觉不太对" / "好像有问题"

**否定模式**:
- 连续两次提出同类修改请求
- Agent 输出后回复短句（<10 字）且不含肯定词
- 包含 "?" 且质疑 Agent 输出正确性
